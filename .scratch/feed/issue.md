## Problem Statement

木下需要一个「备忘录感」的个人内容发布空间——写完即发、不编辑、不纠结。内容有两种形态：原创碎片文字（碎碎念）和网页收藏（剪藏）。当前两者没有统一的发布和展示入口，也无法以时间线形式对外呈现。

木下从手机（华为鸿蒙/小米）和电脑两端发布。发布后有轻度管理需求（隐藏/删除），但不做复杂的编辑和草稿系统。

## Solution

在 catstarry.xyz 上建造 /feed 板块：统一时间线页面，公开浏览碎碎念和剪藏混合内容；右下角浮动 + 按钮触发发布面板，支持文字、图片（最多 6 张 / &le; 20MB）、视频（&le; 100MB / &le; 1 分钟）。剪藏输入 URL 自动抓取 Open Graph 元数据生成链接卡片。

## 定向回流覆盖规则（2026-07-15）

- Feed 是唯一的公开足迹／来时路：统一展示原生碎碎念、剪藏，以及 Blog 发布、Learn 小节完成、Projects 实质更新三类系统足迹。
- 原生发布面板仍只创建碎碎念或剪藏；系统足迹由受控生产者创建，不能手动伪造。
- Blog 仅在首次生产部署成功后产生足迹；Learn 仅在明确完成小节后产生足迹；Projects 仅在木下显式标记实质更新后产生足迹。
- 普通编辑、保存、部署失败、重复部署、草稿、隐藏来源与历史内容不产生新足迹；不做历史回填。
- 系统足迹保存创建时快照，可在 Feed 隐藏但不影响来源内容；来源删除或链接失效不删除其足迹。
- 以下旧文本若仅描述“碎碎念 + 剪藏”时间线，应按本节扩展解释；Home 不再消费 Feed 内容。

数据存储于 Cloudflare D1（帖子）和 R2（媒体文件），通过 feed-api Worker 读写。认证用用户名密码（与发布共用），管理后台独立页面。

## User Stories

1. 作为木下，我想在手机上点右下角的 + 按钮，快速写一段文字并发布，这样我的碎片想法不会丢失。
2. 作为木下，我想发布碎碎念时附带图片（最多 6 张）或一个短视频，让内容更丰富。
3. 作为木下，我想粘贴一个链接后自动生成标题和摘要，只需补充一句点评就能发布剪藏。
4. 作为木下，我发布后内容立即出现在时间线，不需要额外操作或审批。
5. 作为木下，我发布后不想再编辑——要么隐藏（仅我可见），要么直接删除。
6. 作为木下，我想在管理后台按状态、类型、日期筛选帖子，批量隐藏或删除。
7. 作为木下，我想发布面板有草稿暂存功能（12 小时），这样切出 App 再回来不会丢失输入。
8. 作为访客，我想在 /feed 页面看到木下的碎碎念和剪藏混合时间线，按时间倒序排列。
9. 作为访客，我想看到碎碎念卡片包含文字、图片网格或视频封面。
10. 作为访客，我想看到剪藏卡片包含标题、摘要、封面缩略图、木下的点评，点击标题跳到原文。
11. 作为访客，我在手机和平板上浏览时间线时阅读体验舒适。
12. 作为 cati（木下的伴侣），我看到的 /feed 页面和访客一样，无需区分权限。
13. 作为木下，我发布的内容不需要字数/格式限制，写多少发多少。

## Implementation Decisions

### 架构决策

- **渲染模式**：SSR。时间线页面在请求时从 D1 动态拉取，保证发布后立即可见；不做 SSG。
- **数据存储**：D1 存储帖子元数据和文字内容（feed_posts 表）；R2 存储媒体文件。
- **API**：扩展现有 feed-api Worker（目前只有 /api/views），新增 /api/feed 路由。
- **前端**：发布面板为 React island（client:load），时间线页面为 Astro SSR 页面。

### D1 表结构

feed_posts 表：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | TEXT PK | UUID |
| type | TEXT | "note"（碎碎念）或 "clip"（剪藏） |
| content | TEXT | 文字内容 |
| media_json | TEXT (JSON) | R2 对象 key 数组，如 ["feed/2024/abc123.jpg"] |
| link_url | TEXT | 剪藏目标 URL |
| link_title | TEXT | 剪藏标题（自动抓取或手动输入） |
| link_summary | TEXT | 剪藏摘要 |
| link_image | TEXT | 剪藏封面图 URL |
| status | TEXT | "public" / "hidden" / "deleted" |
| created_at | TEXT | ISO 8601 时间戳 |

### 发布面板

- 右下角浮动 + 按钮（position: fixed），点击展开面板。
- 面板默认碎碎念模式，可切换到剪藏模式。
- 碎碎念：textarea → 图片选择/视频选择 → 发布。
- 剪藏：URL 输入 → 自动抓取 Open Graph → 用户评论（可选）→ 发布。
- 发布后关闭面板，时间线刷新。
- 未登录时按钮隐藏（UI 占位）。
- 移动端：底部弹出式面板，占屏幕 2/3。

### 图片/视频约束

- 图片：最多 6 张，单张 ≤ 20MB。
- 视频：1 个，≤ 100MB，≤ 1 分钟。
- 一条帖子只能选图或视频其中一种（互斥）。
- 上传到 R2，key 格式：eed/{YYYY}/{uuid}.{ext}。
- 上传过程中显示进度。

### 剪藏预览

- Worker 端实现 Open Graph 抓取（fetch 目标 URL → 解析 HTML → 提取 og:title/og:description/og:image）。
- 抓取超时 5 秒。
- 抓取失败时 fallback 到手动输入标题和摘要。

### 时间线页面

- /feed/ 路由，Astro SSR。
- 从 D1 拉取 status = "public" 的帖子，按 created_at 倒序。
- 两种卡片视觉：
  - 碎碎念：文字 + 图片网格（1 张全宽、2 张小图、3-6 张小网格）+ 视频播放器 + 发布时间。
  - 剪藏：标题（链接） + 摘要 + 封面缩略图 + 用户评论 + 发布时间。
- 发布时间显示：相对时间（"3 小时前"）+ hover/tap 显示绝对时间。
- 空状态："还没有碎碎念，写点什么吧！"。
- 不做分页——滚动加载（无限滚动或"加载更多"按钮）。

### 认证

- 用户名密码登录，与发布共用同一套。
- Session 有效期 12 小时。
- 存储在 KV。
- /login 登录页。
- 认证失败返回 401 + 错误提示。

### 管理后台

- /feed/admin 页面（需认证）。
- 帖子列表，按 status/type/date 筛选。
- 操作：隐藏（标记 hidden）、删除（标记 deleted）。
- 批量操作：多选 → 统一隐藏/删除。
- 搜索：按 content 模糊匹配。
- 归档浏览：按月份折叠。

### 草稿暂存

- 发布面板的文字内容自动暂存，避免切出页面后丢失输入。
- 暂存有效期 12 小时，过期自动清除。
- 每次打开面板时恢复上次暂存内容。
- 具体存储方案（localStorage / KV / D1 等）由 Phase 3 架构设计决定。

### 设计基调

- 继承全站「艺术温暖」视觉风格（暖米白底、暖棕文字、暖橘强调）。
- 时间线单列垂直滚动，阅读宽度 ~680px。
- 卡片之间用细线或阴影分隔，有呼吸感。
- 浮动 + 按钮用暖橘色圆形，与全站强调色一致。

## Testing Decisions

- **最高接缝**：feed-api Worker 的 HTTP 端点（GET/POST /api/feed）。验证 CRUD 操作和响应的 JSON schema。
- **页面接缝**：/feed/ 页面的 SSR 输出 HTML。验证包含帖子卡片、空状态处理。
- **组件接缝**：发布面板 React island 的交互行为——输入→发布→面板关闭→时间线刷新。
- **先验参考**：项目中 /blog 板块的静态分析验收模式。feed 为 SSR，需额外验证 Worker 端点的响应正确性。
- **测试哲学**：只测外部可观测行为，不测 Astro/React 内部实现细节。

## Out of Scope

- 评论和点赞功能——不做。feed 是广播模式，不是社交平台。
- RSS feed for /feed——不做。只做 /blog 的 RSS。
- 全文检索——时间线的无限滚动 + 管理后台的关键词搜索已覆盖。
- 定时发布/排期——不做。
- 多用户发布——木下唯一发布者。
- Live Photo 支持——格式不统一（小米/苹果 heic、华为不同），暂时取消。
- 发布后编辑功能——与「备忘录感」哲学冲突，不做。
- 通知推送——不做。

## Further Notes

- /feed 是公开足迹的唯一时间流；Home 不再聚合或消费其内容。
- feed-api Worker 同时承载阅读量 API 和 feed API，注意路由隔离和错误处理。
- 图片/视频存储使用 R2，确保 CORS 配置允许前端直传或通过 Worker 代理上传。
- 与 /blog 的关系：两者独立；Blog 首次生产部署成功后向 Feed 写入一条 Public Footprint。
