## 目标

在现有 `workers/feed-api` 上建立 Public Timeline 数据接缝：原生 `feed_posts` 支持碎碎念/剪藏 CRUD；独立 `public_footprints` 保存 Blog 发布、Learn 小节完成、Projects 实质更新的不可变系统足迹。`/api/feed` 统一读取两类公开记录。

## 验收标准

- [ ] D1 新增或确认 `feed_posts` 表，schema 包含：
  - `id` TEXT PRIMARY KEY（UUID）
  - `type` TEXT NOT NULL（`note` | `clip`）
  - `content` TEXT（正文文字）
  - `media_json` TEXT（JSON 数组，存储 R2 文件 key 列表）
  - `link_url` TEXT（剪藏原链接）
  - `link_title` TEXT（剪藏标题，来自 og:title）
  - `link_summary` TEXT（剪藏摘要，来自 og:description）
  - `link_image` TEXT（剪藏封面图，来自 og:image）
  - `visibility` TEXT NOT NULL DEFAULT `public`（`public` | `private`）
  - `created_at` TEXT NOT NULL（ISO 8601）
  - `updated_at` TEXT NOT NULL（ISO 8601）
- [ ] `POST /api/feed` — 仅创建原生帖子，接收 body `{ type, content, media?, link_url?, link_title?, link_summary?, link_image? }`；不得手动创建系统足迹
- [ ] D1 新增 `public_footprints`：包含 `source_module`、`source_ref`、`source_version`、`event_type`、`snapshot_json`、`occurred_at`、`visibility`、唯一 `idempotency_key`
- [ ] Worker 内部提供 `recordFootprint(candidate)` 写入接缝；只允许 Blog 部署成功、Learn 明确完成小节、Projects 显式实质更新的受控生产者调用，不新增面向访客的通用写入 API
- [ ] `GET /api/feed?limit=20&cursor=xxx` — 统一查询 `feed_posts` 与 `public_footprints` 的公开记录，按 `(occurred_at, id)` 倒序分页，返回带明确 `kind` 的时间线条目
- [ ] 所有 API 响应包含 CORS 头（`Access-Control-Allow-Origin: *`、`Access-Control-Allow-Methods: GET, POST, PATCH, DELETE, OPTIONS`、`Access-Control-Allow-Headers: Content-Type, Authorization`）
- [ ] `wrangler deploy` 成功，`curl` 测试通过
- [ ] 复用现有 feed-api Worker 的 `env.DB`（D1 binding）和 `env.VIEW_KV`（KV binding）

## 技术要点

- 表名使用 snake_case
- 游标分页：统一使用 `occurred_at` + `id` 复合游标；原生帖子读取投影为 `occurred_at = created_at`，避免 offset 分页
- 系统足迹写入使用不可变快照与 `idempotency_key`；来源删除或链接失效不回写/删除足迹，隐藏足迹不影响来源内容
- 新增 schema 通过 `wrangler d1 execute feed-db --file=schema.sql --remote` 执行
- 复用现有 Worker 的 `src/index.ts` 路由结构（`/api/views` 已存在，新增 `/api/feed` 路由）
