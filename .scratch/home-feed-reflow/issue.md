# Home / Feed 定向回流 — Product Requirement Document

> **阶段**：定向 Phase 2（规格化）
>
> **触发原因**：Phase 4.1 已确认的 Home 设计变更。

---

## Problem Statement

原 Home 被定义为五个来源的混合时间线，并依赖聚合 API、筛选器与跨模块排序。这与已确认的设计方向冲突：Home 应是进入 catstarry 宇宙的入口和星图索引，而不是内容流。

同时，Blog 发布、Learn 完成与 Projects 实质更新缺少一个统一、公开、克制的“来时路”。这些信号应该出现在 Feed，而不是通过 Home 聚合展示。

## Solution

建立三个协同的产品边界：

1. **Home 星图入口**：宇宙入口 → 接近星域 → 自由星图总览 → 页脚。Home 不展示混合时间线。
2. **Feed 公开足迹**：在原有碎碎念、剪藏之外，展示 Blog 发布、Learn 小节完成、Projects 实质更新。
3. **Public Footprint 契约**：来源模块仅在发布、完成或显式标记的实质更新时，创建一个不可随普通编辑改写的公开足迹。

About 是 Home 的例外：它在星图中原地展开，不跳转为独立信息流。

## User Stories

1. 作为访客，我打开 Home 时想进入木下的宇宙入口，而不是先看到一串混合内容卡片。
2. 作为访客，我滚动接近同一片星域后，想看到自由分布的 Blog、Feed、Learn、Projects 与 About 星球。
3. 作为访客，我点击 Blog、Feed、Learn 或 Projects 星球时，想经过短暂推进后进入对应的功能页面。
4. 作为访客，我点击 About 星球时，想在 Home 原地阅读介绍，而不离开星图。
5. 作为访客，我想在 Feed 看到木下的原生碎碎念、剪藏和重要创作足迹，以时间顺序理解其来时路。
6. 作为访客，我想明确区分一条内容是手写碎碎念、剪藏，还是 Blog 发布、Learn 完成或 Projects 更新。
7. 作为木下，我发布 Blog 后只想在生产部署成功时看到一条 Feed 足迹，而不因提交或重试产生重复记录。
8. 作为木下，我完成 Learn 的一个小节时，想显式标记完成后才产生足迹，不让普通笔记编辑干扰时间流。
9. 作为木下，我更新 Projects 时，想由自己显式标记“实质更新”，而不让系统猜测。
10. 作为木下，我想隐藏某条系统足迹而不影响原 Blog、Learn 或 Projects 内容。
11. 作为木下，我不想回填旧内容；Feed 只从这一机制上线后记录新事件。

## Implementation Decisions

- **唯一跨模块语义缝**：Public Footprint。它描述用户可见事件，不预设底层复用现有帖子存储还是建立独立事件存储；该决定留给定向 Phase 3。
- **Home 的责任**：叙事、星图导航、About 原地展开；不读取或聚合 Blog、Feed、Learn、Projects 的内容数据。
- **Feed 的责任**：唯一公开事件流；混排原生内容与系统足迹。
- **足迹触发**：
  - Blog：生产部署成功。
  - Learn：明确完成一个小节。
  - Projects：木下显式标记实质更新。
- **足迹快照**：创建时保存标题、摘要、链接、时间及来源标识；普通源内容编辑不回写足迹。
- **去重**：同一来源版本只能产生一条足迹；重复部署、普通编辑、重新保存不产生新足迹。
- **可见性**：系统足迹可从 Feed 隐藏；隐藏不影响源内容。
- **发布边界**：原生 Feed 发布面板仍只创建碎碎念或剪藏，不能手动创建系统足迹。

## Testing Decisions

- Home 验证只覆盖可见的入口、滚动阶段、星图导航与 About 原地展开；不验证内容聚合，因为该行为已退出范围。
- Feed 验证按用户可见时间线检查五类系统/原生事件的顺序、来源标识、链接、隐藏行为和重复抑制。
- 触发验证以外部结果为准：成功部署、完成小节、显式项目更新各产生一次足迹；普通编辑、部署失败和重复部署不产生足迹。
- 不测试内部表结构、Worker 实现细节或前端框架内部状态。

## Out of Scope

- 历史 Blog、Learn、Projects 或 Poker 项目的回填。
- Home 上的 Recently 区块、混合时间线、类型筛选、分页或无限滚动。
- 在本轮决定星球的最终视觉、材质、鼠标流星、豹猫卫星或具体动画实现。
- 把 Blog、Feed、Learn、Projects 功能页面重做成星球页面。
- 让系统自动推断 Projects 是否属于实质更新。

## Further Notes

- 旧的 Home 聚合 API、HomeTimelineItem 和相关缓存策略需要在定向 Phase 3 通过架构复核替换或废弃。
- 本 PRD 只定义产品行为；事件存储、生产者接口、部署成功信号和迁移策略属于下一阶段的架构决定。
